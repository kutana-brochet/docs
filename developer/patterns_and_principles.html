K+B Office Add-ins

Patterns and Principles

Caution

Adhere to Agreed Approaches Consistency is our primary driver for velocity. Do not change architectural approaches (e.g., swapping CQS for Services) without discussing it with the team first. Least Surprise > Clever Solutions.

1. Core Philosophy

Least Surprise & Consistency

We value code that looks boring. If you solve a problem in Module A, the solution in Module B should look identical.

Tell-Don't-Ask

Encapsulate logic within your Domain Models. Don't pull data out to process it; ask the object to do the work.

❌ Bad: Anemic Model

var user = _repo.GetUser(id);
if (user.IsActive && user.Expiry > DateTime.Now) {
    user.LastLogin = DateTime.Now; // Logic leaks into the service/handler
    _repo.Save(user);
}

✅ Good: Rich Model

var user = _repo.GetUser(id);
var result = user.Login(); // Logic stays in the User entity
if (result.IsSuccess) _repo.Save(user);

2. Architecture: Where Does This Code Belong?

We follow a rigid dependency hierarchy. Use this decision tree:

Is it a business rule or invariant?
├── Yes → Domain (Entity or Value Object)
│   └── Does it need identity tracking?
│       ├── Yes → Entity
│       └── No → Value Object
└── No → Continue...

Is it orchestrating multiple domain objects?
├── Yes → Application (Command/Query Handler)
└── No → Continue...

Is it calling external systems (DB, API, files)?
├── Yes → Infrastructure
└── No → Continue...

Is it translating user input to commands?
├── Yes → Presentation (VSTO/Ribbon)
└── No → Re-evaluate. Something's off.

Layer Summary

Layer Contains Dependencies Examples
Domain Entities, Value Objects, Domain Events None EntitlementType, Template, Script
Application Commands, Queries, Handlers, Interfaces Domain CreateDocument, GetScripts, IUserNotifier
Infrastructure Implementations, DB, File System Application, Domain TemplateRepository, CommandDispatcher
Presentation Ribbons, UI Application, Domain Thin dispatch only

Quick Checks

  • Domain has no dependencies → If you're adding a using for infrastructure, wrong layer
  • Application orchestrates, doesn't decide → Business logic in handler = anemic domain smell
  • Infrastructure implements interfaces → Defined in Application, implemented in Infrastructure
  • Presentation is trivial → Just dispatch commands/queries, no logic

Anti-patterns

  • ❌ if (user.IsActive && user.HasPermission) in a handler → belongs in user.CanPerform()
  • ❌ Repository interface in Infrastructure → belongs in Application
  • ❌ COM objects in Application layer → stays in VSTO project

3. BDD: Before You Code

Decision prompt:

"Do I have a Gherkin scenario that describes what I'm building?"

Answer Action
No Write the scenario first. No exceptions.
Yes, but vague Refine until Given/When/Then are concrete and testable
Yes, specific Proceed to red → green → refactor

Heuristics

  • Scenario should be writable without knowing implementation
  • If you can't write the scenario, you don't understand the requirement
  • One scenario = one behavior (not one feature)

Anti-patterns

  • ❌ Writing tests after code "to get coverage"
  • ❌ Scenarios that describe UI clicks, not behavior
  • ❌ Multiple When clauses (that's multiple scenarios)

4. CQS: Command or Query?

Decision prompt:

"Does this operation change state?"

Answer Type Returns
Yes Command Result or Result<T>
No Query Result<T> with data, no side effects

Commands Can Read Internally

From CreateDocument:

[RequireEntitlement(EntitlementType.WordDocumentManagement)]
[RequireEntitlement(EntitlementType.WordStandardTools)]
public class Handler(
    ILogger<Handler> logger,
    ITemplateRepository templateRepository,           // reads
    IDocumentManagementService documentManagementService,
    IDocumentRepository documentRepository,
    IUserNotifier userNotifier,                       // notifies
    IApplicationEventDispatcher applicationEventDispatcher)
    : ICommandHandler<Command>
{
    public Result Handle(Command command, CancellationToken cancellationToken = default)
    {
        var result = templateRepository.GetByTag(command.Value);  // internal read

        if (!result.HasValue)
        {
            userNotifier.Alert(error, "Template Error");          // notify on failure
            return Result.Failure(error);
        }

        var doc = documentManagementService.CreateDocument(result.Value);  // mutate

        applicationEventDispatcher.DispatchAsync(
            new DocumentCreatedFromTemplateEvent(doc.Value, template));    // event

        return doc;
    }
}

Queries Stay Pure

From GetScripts:

public class Handler(IScriptRepository repository) : IQueryHandler<Query, IEnumerable<Script>>
{
    public Result<IEnumerable<Script>> Handle(Query query, CancellationToken cancellationToken = default)
    {
        return Result.Success(repository.GetScripts());  // read only
    }
}

Rules

  • Commands read what they need internally
  • IUserNotifier in command handler keeps notification testable
  • Queries have no side effects
  • Use domain events for cross-aggregate coordination

Anti-patterns

  • ❌ Separate query dispatch just to read data for a command
  • ❌ Query that writes audit logs → use decorator or event
  • ❌ Notification logic in Ribbon → untestable

5. Design Principles (SOLID)

Single Responsibility (SRP)

Handlers should do one thing.

❌ Bad

UpdateUserHandler: Validates input, checks DB, updates User, sends Email, logs to file.

✅ Good

UpdateUserHandler: loads User, calls user.Update(), saves. Separately: UserUpdatedEvents trigger a SendEmailEventHandler.

Open/Closed (OCP)

Extend behavior without modifying existing code.

❌ Bad

if (type == "Word") ExportWord();
else if (type == "PDF") ExportPdf();
// Modifying this method every time we add a format.

✅ Good

// Inject IEnumerable<IExportStrategy>
_strategies.First(s => s.Supports(type)).Export();
// Add new class PdfExportStrategy : IExportStrategy. No change to caller.

Liskov Substitution (LSP)

Subtypes must be substitutable for their base types.

❌ Bad

public class ReadOnlyList : IList -> throws NotSupportedException on Add(). Why? code expecting an IList assumes it can Add.

✅ Good

Use IEnumerable or IReadOnlyCollection if the list shouldn't be modified.

6. Coding Standards

Naming Conventions

Remove ambiguity.

  • Commands: <Verb><Noun> → CreateDocument, RunScript, ActivateLicence
  • Queries: Get<Noun> / Find<Noun> → GetScripts, GetLicence

❌ Bad

  • DocumentManager (What does it do?)
  • ProcessData (How?)

✅ Good

  • ITemplateRepository
  • SaveDocument

Logging

Log for the poor soul debugging this at 3 AM.

❌ Bad

catch (Exception ex) {
    _logger.LogError("Error happened"); // What error? What context?
}

✅ Good

catch (Exception ex) {
    _logger.LogError(ex, "Failed to create document from template {TemplateName}", template.Name);
}

When to Log:

  • Info: Significant business events ("Document created successfully").
  • Warning: Recoverable issues ("File locked, retrying").
  • Error: Exceptions/Failures that stop the flow.

Commits

Follow Conventional Commits.

❌ Bad

  • "fixed bug"
  • "updates"
  • "wip"

✅ Good

  • fix(templates): handle missing template gracefully
  • feat(scripting): add custom property commands
  • chore: update nuget packages

7. Specific Patterns

Factories vs DI

❌ Bad

Creating a factory just to call new.

public class ServiceFactory {
    public Service Create() => new Service(); // Just register Service with DI!
}

✅ Good

Runtime decisions.

public IExporter Create(string format) {
    return format switch {
        "PDF" => new PdfExporter(),
        "Word" => new WordExporter(),
        _ => throw new NotSupportedException()
    };
}

Records vs Classes

Type Use Case Rule
Record DTOs, Messages, Config Use when data is the value. new Point(1,2) == new Point(1,2).
Class Domain Entities Use when identity matters.
