Naming Convention Migration Analysis

Current codebase vs Clean Architecture Naming Cheatsheet

Executive Summary

The codebase is ~75% aligned with the cheatsheet. The architecture layers, dependency direction, and most suffix conventions already match. The gaps are primarily structural naming (folder names, config suffixes) and feature area organisation (missing Features/ wrapper, Abstractions/ vs Common/).

Differences

1. Application Layer — Missing Features/ Wrapper

Aspect Current Cheatsheet
Feature areas Core.Application/Document/, Core.Application/Licensing/ Core.Application/Features/Documents/, Core.Application/Features/Licensing/

Feature slices sit directly under Core.Application/. The cheatsheet expects a Features/ grouping folder to separate use-case code from cross-cutting Common/ and Abstractions/.

Impact: Medium. Makes it harder to distinguish feature code from infrastructure at a glance.

2. Abstractions/ vs Common/ Split

Aspect Current Cheatsheet
Cross-cutting interfaces Abstractions/ (Behaviours, Messaging, Localisation, Notifications) Common/ (Behaviours, Interfaces, Messaging, Models, Exceptions)
Shared utilities Common/ (LoopHelpers, Messages, Errors, RuleEvaluation) Also Common/

The codebase has two top-level folders (Abstractions/ + Common/) where the cheatsheet has one (Common/). Content overlaps — both hold cross-cutting concerns.

Impact: High cognitive load. Developers must guess which folder to check.

3. Feature Area Naming — Singular vs Plural

Current Cheatsheet
Document/ (singular, Application) Documents/ (plural)
Documents/ (plural, Domain) Documents/ (plural)
DocumentManagement/ (Application) Would be a sub-area or merged into Documents/

The cheatsheet mandates plural nouns for feature area folders. The codebase mixes singular (Document/) and compound names (DocumentManagement/).

Impact: Low-medium. Inconsistency, not a structural problem.

4. Configuration Naming — *Configuration vs *Options/*Settings

Current Cheatsheet
14x *Configuration classes *Options (raw POCO) + *Settings (typed accessor)
All implement IUserConfiguration *Options bound from config; *Settings wraps IOptionsMonitor<*Options>
No I*Settings application interfaces Handlers depend on I*Settings, not config POCOs

This is the largest naming gap. The cheatsheet separates concerns:

  • *Options = dumb POCO bound from JSON/registry
  • *Settings = smart wrapper implementing an Application-layer I*Settings interface

The current *Configuration pattern conflates both roles.

Impact: High. Affects dependency direction (handlers reference Infrastructure types).

5. Domain — Word/ Namespace

Current Cheatsheet
Core.Domain/Word/ (DocumentPath, Templates, ContentControls) Feature areas only (Documents/, Licensing/) — no host-app namespace

The cheatsheet avoids host-app namespaces in Domain. DocumentPath, Template, etc. would live under their feature area (e.g., Documents/ or Templates/).

Impact: Medium. Couples Domain to "Word" concept when it should be host-agnostic.

6. Application Document/Interfaces/ Subfolder

Current Cheatsheet
Document/Interfaces/ (20+ interfaces) Co-locate at feature area root OR Common/Interfaces/ if cross-area

The cheatsheet doesn't use an Interfaces/ subfolder within feature areas. Interfaces are either:

  • At feature area root (feature-specific)
  • In Common/Interfaces/ (cross-cutting)

Impact: Low. Organisation preference, but creates deep nesting.

7. AddIn Structure — Missing Startup/ Folder

Current Cheatsheet
DI in ThisAddIn.cs + Infrastructure.Common/DependencyInjection.cs Startup/ContainerConfig.cs + Startup/AddInBootstrapper.cs
No Startup/ folder Dedicated startup folder

DI registration is split between ThisAddIn.cs (300+ lines of setup) and the Infrastructure DependencyInjection extension methods. The cheatsheet extracts this to a Startup/ folder.

Impact: Medium. ThisAddIn.cs is overloaded with registration logic.

8. AddIn Services/ Folder

Current Cheatsheet
WordAddIn/Services/ (AddInInfoService, EnvironmentService, XmlDocumentInspectorService) No Services/ — these would be in Interop/ or feature folders

The cheatsheet doesn't have a Services/ folder in the AddIn. Service implementations belong in Interop/ (if COM-dependent) or Infrastructure.

Impact: Low. Only 3 files.

9. Missing Mapper vs Adapter Distinction

Current Cheatsheet
WordContentControlMapper (correct) Mapper = pure data transform, Adapter = interface bridging
DmsProfileConverter Would be DmsProfileMapper or DmsProfileAdapter
WordContextAdapter (correct)

Some classes use Converter suffix where the cheatsheet would use Mapper or Adapter.

Impact: Low. Naming clarity.

10. ComInterop/ vs Interop/

Current Cheatsheet
Both ComInterop/ and Interop/ in WordAddIn Single Interop/ folder

ComInterop/ contains the COM-visible interface (IComInterface, ComOperations). The cheatsheet uses a single Interop/ folder.

Impact: Low. Could argue ComInterop/ is a valid cohesive subfolder per the "3+ classes" rule.

Decision Points

These are choices where multiple valid approaches exist and the team needs to decide before migrating.

D1: Introduce Features/ Wrapper or Keep Flat?

Option A — Add Features/: Matches cheatsheet. Clear separation of feature code vs cross-cutting. Option B — Keep flat: Less nesting. Feature areas already have distinct names. Trade-off: Readability vs brevity. With 10+ feature areas the Features/ wrapper aids navigation.

D2: Merge Abstractions/ into Common/ or Keep Both?

Option A — Merge into Common/: Single place for cross-cutting code. Matches cheatsheet. Option B — Keep Abstractions/: Some teams prefer separating interfaces from utilities. Trade-off: Discoverability (one place to look) vs separation of concerns (interfaces vs helpers).

D3: Rename *Configuration to *Options + Extract *Settings?

Option A — Full migration: Rename all 14 Configuration classes to Options, create I*Settings interfaces in Application, create *Settings implementations in Infrastructure. Option B — Gradual: Only apply to new code; migrate existing as you touch them. Option C — Skip: Current pattern works; migration is high-churn for naming gain. Trade-off: This is the highest-impact change. It fixes dependency direction (handlers shouldn't reference Infrastructure config types) but touches ~40+ files.

D4: Flatten Domain/Word/ into Feature Areas?

Option A — Move to feature areas: DocumentPath → Documents/, Template → Templates/, etc. Option B — Keep Word/: The codebase serves Word specifically; the namespace is honest. Trade-off: Purity (host-agnostic domain) vs pragmatism (it IS a Word add-in). If PowerPoint shares these types, flattening makes more sense.

D5: Dissolve Document/Interfaces/ Subfolder?

Option A — Promote to feature area root: Move 20+ interfaces up one level. Option B — Promote cross-cutting ones to Common/Interfaces/: Keep feature-specific at root, move shared ones. Option C — Keep subfolder: 20 interfaces at root would be noisy. Trade-off: Flat-but-noisy vs nested-but-organized. With 20+ interfaces, the subfolder might be the pragmatic choice despite the cheatsheet preference.

D6: Extract Startup/ from ThisAddIn.cs?

Option A — Extract: Create Startup/ContainerConfig.cs and Startup/AddInBootstrapper.cs. Option B — Keep inline: DependencyInjection.cs already handles most of it. Trade-off: ThisAddIn.cs is 425 lines. Extracting startup would bring it under 200 and match the cheatsheet.

D7: Rename Document/ to Documents/ (Plural)?

Option A — Rename: Consistent plural nouns across all feature areas. Option B — Keep: Document is also a valid feature area name (it's a bounded context, not a collection). Trade-off: Consistency vs namespace change churn. Affects imports across the entire solution.

Step Files Affected Risk

Step Files Affected Risk

Step Files Affected Risk

What Already Matches

These are areas where the codebase already follows the cheatsheet:

  • Project structure & dependency direction (Domain → Application ← Infrastructure ← AddIn)
  • Feature slice pattern (static wrapper with nested Command/Query/Handler)
  • Decorator chain in Behaviours/ with explicit ordering
  • Repository interfaces co-located in Domain (IDocumentRepository, ILicenceRepository, ITemplateRepository)
  • Event handler naming (DocumentBeforeSaveEventHandler pattern → AttachTemplateEventHandler, etc.)
  • Interop wrapper naming (WordDocumentRepository, WordContentControlFinder, WordObjectModel* prefix)
  • DomainEvents subfolder in Domain feature areas
  • Application events in feature area Events/ folders
  • BDD test structure with .feature + StepDefinitions pairs
  • Infrastructure DI extension methods (AddCoreInfrastructure())
  • Owned vs External state distinction (Licence as aggregate, Document as projection)
