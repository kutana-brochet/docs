Coding Guidelines (BDD + DDD Aligned)

  1. Purpose A concise, copy-paste-ready guide for adding or modifying behavior using Behavior-Driven Development (BDD) and Domain-Driven Design (DDD) in this solution (Core.Domain, Core.Application, Infrastructure, AddIns).
  1. Core Principles • Let behavior drive design: Feature (Gherkin) → Failing Scenario → Minimal Code → Refactor. • The model reflects ubiquitous language; rename mercilessly when language evolves. • Keep business logic inside domain entities/value objects where it naturally fits. • One reason to change per class (SRP). • Prefer explicit over implicit; no “utility” dumping grounds. • Avoid premature abstractions; extract only after clear duplication.
  1. BDD Workflow (Always)
  2. Write/extend a .feature file (Reqnroll).
  3. Define Scenario(s) in Gherkin (Given/When/Then) using domain language.
  4. Generate only missing step definitions.
  5. Run: confirm scenario(s) fail (RED).
  6. Implement smallest domain/application change (GREEN).
  7. Refactor: improve names, enforce invariants, collapse duplication.
  8. Commit: feat(template): auto-attach default when opening untemplated document.

Example:

Feature: Auto attach default template
  Scenario: Attaching default template when enabled
    Given auto attach template is enabled
    And a document without a template is opened
    When the document open workflow runs
    Then the default template "Normal.dotm" is attached
    And the attachment is recorded in custom properties
  1. Architecture (Layer Responsibilities) | Layer | Purpose | Contains | Depends On | |-------|---------|----------|------------| | Core.Domain | Business model & invariants | Entities, Value Objects, Domain Events, Domain Services, Repository Interfaces | None | | Core.Application | Use cases & orchestration | Application Services, Commands/Queries, DTOs, Event Dispatch | Core.Domain | | Infrastructure.* | Technical implementations | Repos, persistence, integrations, file system, configuration | Core.Domain, Core.Application | | WordAddIn | Office entry point & UI adapters | Startup, DI composition, Ribbon/UI, OOM adapters | All (composition root) | | Tests | Specification & regression safety | Feature (.feature), Unit, Integration tests | All necessary | Dependency Rule: Flow inward only (UI → App → Domain). No reverse references.
  1. DDD Tactical Patterns
  • Entities
    • Identity-based (e.g., Document, Template).
    • Enforce invariants internally.
    • Expose intention-revealing methods (e.g., AttachTemplateIfMissing()).
  • Value Objects
    • Immutable; equality by all components.
    • Replace primitive clusters (e.g., consider TemplateVersion, TemplatePath if rules grow).
  • Aggregates
    • Small, cohesive transactional boundary.
    • One repository per aggregate root.
    • External aggregates referenced by ID (not object graph).
  • Domain Services
    • Stateless, rare. Use only when logic spans multiple aggregates and does not belong to one.
  • Domain Events
    • Past tense (TemplateAttached).
    • Immutable data; used to signal side-effects or consistency policies.
  • Factories
    • Encapsulate complex creation (validate before expose). Static Create acceptable for simple cases (already used in Template.Create).
  1. Code Placement Decision (Ask in Order)
  2. Is this a business rule or invariant? → Core.Domain.
  3. Is it orchestration/use-case flow? → Core.Application.
  4. Is it persistence, file system, OOM, configuration? → Infrastructure or AddIn adapter.
  5. Is it Office-specific interaction? → AddIn.
  6. Only test wiring/state? → Test project. If you’re modifying TemplateRepository: • Keep it pure infrastructure (materializing Template). • Do NOT embed business policy (e.g., when to attach). That belongs in an application service or domain method invoked by a use case.
  1. Naming & Language • Reflect domain terms precisely (no “Process”, “HandleData”, “Util”). • Methods narrate intent (e.g., SelectAttachableTemplates(), not GetList). • Centralize repeated constants (e.g., default template name) via configuration or value object.
  1. Office / VSTO Guidelines • Contain Word/Office Object Model (OOM) interactions inside AddIn/adapters. • Never pass COM types into Domain/Application layers. • Release COM objects deterministically. • Translate OOM failures to domain/application friendly outcomes (avoid leaking COMException outward).
  1. Error & Result Handling • Use Result pattern for expected failures (already using CSharpFunctionalExtensions). • Throw domain exceptions only for invariant violations—not control flow. • Never surface raw infrastructure/COM exception details beyond boundary. • Log with context but not sensitive data.
  1. Testing Strategy | Test Type | Focus | Examples | |-----------|-------|----------| | Feature (BDD) | Cross-layer behavior | Auto attach default template | | Unit (Domain) | Invariants & rules | Template naming/validation | | Unit (Application) | Orchestration | AttachTemplateOnOpen handler | | Integration | Repo/materialization | TemplateRepository parsing | | Smoke/AddIn | OOM wiring | Startup & DI boot correctness | Rules: • One assertion concept per test. • Assert outcomes (state/events), not implementation (no private field assumptions). • Step definitions stay thin: delegate to helpers/use cases.
  1. Refactoring Checklist (After Green) • Are invariants enforced at construction or mutator boundaries? • Any duplicated regex or filtering logic? Extract meaningfully. • Are public methods telling a story? • Any creeping primitive obsession? Promote to Value Object. • Any policy logic sitting in repositories/UI? Move inward.
  1. Anti-Patterns (Block Immediately) • Anemic Domain Model (no behavior on entities). • Putting conditional business logic into repositories (e.g., filtering decisions beyond raw retrieval). • Reusing mutable primitives for domain concepts (dates, IDs, versions). • “Utility” classes with mixed, unrelated methods. • Sprawling aggregates (too many responsibilities). • Tests asserting internal structure instead of observable behavior.
  1. Performance Notes • Only optimize hot paths after profiling (Debug > Performance Profiler). • Avoid unnecessary allocations in repository enumeration (current parallel bag usage is fine; reassess if contention appears). • Encapsulate regex compilation if reused heavily.
  1. Security & Robustness • Validate externally sourced input at Application boundary. • Do not log file paths or template contents if sensitive. • Enforce rule checking server-side (never trust UI alone).
  1. Commit Message Pattern Format: feat(template): auto attach default for untemplated documents fix(repository): ignore temporary Word lock files refactor(domain): introduce TemplateVersion value object
  1. Minimal Daily Flow Plan (Story + Scenarios) → Write/Fail Feature → Implement Domain/App → Pass → Refactor → Commit → Repeat. Boring, repeatable, safe.
  1. Quick Start (Template Example)
  2. Add scenario for selecting attachable templates.
  3. Run → fail.
  4. Ensure Template exposes any new rule (not repository).
  5. Keep TemplateRepository limited to discovery/parsing.
  6. Add application service if orchestration grows.
  7. Refactor names to match new ubiquitous language additions.

Maintain this as a living document—update when domain language or architecture strategy evolves. Done. Let behavior lead design.
