Scripts - Developer Guide

Technical implementation guide for the scripting subsystem, covering architecture, domain model, configuration, and extension points.

Architecture Overview

The scripting subsystem implements a Command Executor pattern enabling users to define document formatting and editing scripts through JSON configuration. Scripts are composed of immutable command objects that are executed sequentially against Word documents via VSTO COM interop.

Architectural Pattern Evolution

The current architecture represents a significant refactoring from the original design (October 2025). See ADR: Refactor Scripting to Command Executor Pattern for the full decision context.

Previous Architecture (deprecated):

IScriptCommand.Execute() → IScriptExecutor → DocumentScriptExecutor (adapter) → IDocumentEditor → Word COM

Current Architecture (dispatcher pattern with individual handlers):

IScriptCommand (pure data) → IScriptCommandDispatcher.Execute() → Dynamic dispatch → IScriptCommandHandler<T> → Word COM

Key Architectural Principles

  1. Pure Data Commands: IScriptCommand implementations are immutable records with no behaviour
  2. Dispatcher Pattern: IScriptCommandDispatcher routes commands to appropriate handlers via dynamic dispatch
  3. Individual Handlers: Each command type has its own IScriptCommandHandler<T> implementation
  4. Backend Agnostic: Design supports multiple backends (Word, Excel, PowerPoint, PDF)
  5. CQRS Integration: Scripts execute via CQRS handlers in the application layer
  6. Clean Architecture: Domain layer has no infrastructure dependencies
  7. Open/Closed Principle: New commands can be added without modifying the dispatcher

Layer Structure

Domain Layer (Core.Domain.Scripting)

Pure domain logic with no external dependencies.

Script Aggregate Root

public class Script : AggregateRoot<string>
{
    public string Name { get; }
    public string Description { get; }
    public IReadOnlyCollection<IScriptCommand> Commands { get; }
}

The Script aggregate is the root entity representing a collection of commands to execute. The Name serves as the aggregate ID.

Invariants:

  • Name must be unique within the repository
  • Commands collection is immutable after construction
  • Command order is preserved

IScriptCommand Interface

Marker interface for all script commands:

public interface IScriptCommand { }

All command implementations are immutable records representing pure data:

public record ApplyFontFormatScriptCommand(
    FontStyle Style,
    FontFormattingRange Range) : IScriptCommand;

public record DeleteCharactersScriptCommand(
    string Characters,
    CommandRange Range,
    CommandSubRange SubRange) : IScriptCommand;

public record ApplyStyleScriptCommand(
    StyleFormatting Style,
    FormattingRange Range) : IScriptCommand;

Design Rationale: Commands contain only data (what to do), whilst executors contain behaviour (how to do it). This separation enables:

  • Serialisation for audit logging
  • Batching for performance optimisation
  • Transformation and composition
  • Backend-agnostic command representation

Command Handler

RunScript: Executes a script's commands sequentially

public class Handler(
    IScriptRepository repository,
    IScriptCommandDispatcher dispatcher,
    ILogger<Handler> logger)
    : ICommandHandler<Command, Result>
{
    public Result Handle(Command command)
    {
        var scriptResult = repository.GetScript(command.ScriptName);
        if (scriptResult.HasNoValue)
            return Result.Failure($"Script {command.ScriptName} not found");

        var script = scriptResult.Value;
        foreach (var scriptCommand in script.Commands)
        {
            var result = dispatcher.Execute(scriptCommand);
            if (result.IsFailure)
            {
                logger.LogError("Command failed: {Error}", result.Error);
                return result;
            }
        }

        return Result.Success();
    }
}

Key Behaviours:

  • Sequential execution - stops on first failure
  • Logging of command failures
  • Transaction-like semantics (all or nothing)
  • Cancellation token support (via extension)

IScriptCommandDispatcher Interface

Dispatcher interface for routing commands to handlers:

public interface IScriptCommandDispatcher
{
    Result Execute(IScriptCommand command);
}

Design Note: This interface abstracts the backend implementation (Word, Excel, etc.) from the application layer. The dispatcher uses runtime type resolution to invoke the correct IScriptCommandHandler<T> for each command type.

IScriptCommandHandler Interface

Generic handler interface for individual command types:

public interface IScriptCommandHandler<in T> where T : IScriptCommand
{
    Result Handle(T command);
}

Enables:

  • Individual command handlers per type
  • Dependency injection per handler
  • Programmatic reuse of script commands
  • Open/Closed Principle compliance
  • Independent testing of each command type

Persistence (Core.Infrastructure.Persistence)

public class ScriptRepository(
    IOptionsMonitor<ScriptConfiguration> configuration,

Common Infrastructure (Core.Infrastructure.Common)

ScriptCommandDispatcher: Dynamic dispatch implementation

Located in Core.Infrastructure.Common.Messaging, this dispatcher uses runtime type resolution to route commands to their registered handlers:

public class ScriptCommandDispatcher(
    IServiceProvider serviceProvider,
    ILogger<ScriptCommandDispatcher> logger) : IScriptCommandDispatcher
{
    public Result Execute(IScriptCommand command)
    {
        if (command is null)
            throw new ArgumentNullException(nameof(command));

        try
        {
            // Resolve handler type at runtime based on command's actual type
            var commandType = command.GetType();
            var handlerType = typeof(IScriptCommandHandler<>).MakeGenericType(commandType);

            logger.LogDebug("Resolving handler {HandlerType} for command {CommandType}",
                handlerType.Name, commandType.Name);

            // Try to get handler from DI
            var handler = serviceProvider.GetService(handlerType);

            if (handler is null)
            {
                logger.LogWarning("No handler registered for script command type {CommandType}", commandType.Name);
                return Result.Failure($"No handler registered for {commandType.Name}");
            }

            // Dynamic dispatch: CLR resolves to correct Handle(TCommand) method at runtime
            return ((dynamic)handler).Handle((dynamic)command);
        }
        catch (Exception ex)
        {
            logger.LogError(ex, "Error dispatching script command {CommandType}", command.GetType().Name);
            return Result.Failure($"Error dispatching command: {ex.Message}");
        }
    }
}

Key Features:

  • Generic type resolution at runtime
  • Dynamic dispatch to strongly-typed handlers
  • Automatic handler discovery via dependency injection
  • No modification required when adding new command types

VSTO Command Handlers (WordAddIn.Features.Scripting.Handlers)

All script commands are implemented as individual handlers. Example handler:

public class DeleteCharactersScriptCommandHandler(
    ITextReplacer textReplacer,
    ILogger<DeleteCharactersScriptCommandHandler> logger)
    : IScriptCommandHandler<DeleteCharactersScriptCommand>
{
    private readonly Application _wordApp = Globals.ThisAddIn.Application;

    public Result Handle(DeleteCharactersScriptCommand command)
    {
        try
        {
            _wordApp.ScreenUpdating = false;
            _wordApp.System.Cursor = WdCursorType.wdCursorWait;

            // Implementation using Word COM interop
            var result = textReplacer.Replace(
                command.Characters,
                string.Empty,
                command.Range);

            return result;
        }
        catch (COMException ex)
        {
            logger.LogError(ex, "COM error deleting characters");
            return Result.Failure($"Error: {ex.Message}");
        }
        finally
        {
            _wordApp.ScreenUpdating = true;
            _wordApp.System.Cursor = WdCursorType.wdCursorNormal;
        }
    }
}

All Command Handlers (located in src/VSTO/WordAddIn/Features/Scripting/Handlers):

Step 4: Handler Auto-Registration

Handlers are automatically registered via Scrutor in ThisAddIn.cs:

builder.Services.Scan(s => s.FromAssemblyOf<WordScriptCommandHandlerBase>()
    .AddClasses(c => c.AssignableTo(typeof(IScriptCommandHandler<>)), publicOnly: false)
    .AsImplementedInterfaces()
    .WithTransientLifetime());

No manual registration required:

  • Scrutor automatically discovers all classes implementing IScriptCommandHandler<T>
  • Handlers are registered with transient lifetime
  • The dispatcher will automatically find and invoke your new handler

private Script MapToScript(string name, ScriptDto dto)
{
    var commands = dto.Commands.Select(cmd => cmd switch
    {
        {CommandName}ScriptCommandDto c => new {CommandName}ScriptCommand(
            c.{Property},

        // ... other command types

    return new Script(name, dto.Comment, commands.ToList());
}

COM Safety and Error Handling

Critical VSTO Practices

1. COM Exception Handling

Always wrap Word API calls:

try
{
    // Word COM operations
}
catch (COMException ex)
{
    logger.LogError(ex, "COM error");
    return Result.Failure($"Error: {ex.Message}");
}

2. COM Object Cleanup

Release COM objects explicitly:

Sentences sentences = null;
try
{
    sentences = range.Sentences;
    // Use sentences
}
finally
{
    if (sentences != null)
        Marshal.ReleaseComObject(sentences);
}

3. Screen Updating and Cursor Management

try
{
    _wordApp.ScreenUpdating = false;
    _wordApp.System.Cursor = WdCursorType.wdCursorWait;
    // Operations
}
finally
{
    _wordApp.ScreenUpdating = true;
    _wordApp.System.Cursor = WdCursorType.wdCursorNormal;
}

4. Infinite Loop Prevention

int iterations = 0;
const int maxIterations = 10000;

while (condition && iterations < maxIterations)
{
    // Operation
    iterations++;
}

if (iterations >= maxIterations)
    logger.LogWarning("Maximum iterations reached");

Performance Considerations

Optimisation Strategies

1. Batch Operations

Process multiple paragraphs/words/sentences in a single operation rather than one-by-one.

2. Minimise Range Access

Cache Range objects rather than repeatedly accessing Selection, Paragraphs, etc.

3. Screen Updating

Disable during bulk operations:

_wordApp.ScreenUpdating = false;
try { /* operations */ }
finally { _wordApp.ScreenUpdating = true; }

4. Command Ordering

  • Apply styles before character formatting
  • Combine formatting commands when possible
  • Use ApplySelectionFormatting for multiple format changes

Known Bottlenecks

  • Paragraph iteration: O(n) for each paragraph access
  • Word/Sentence enumeration: Expensive for large documents
  • Find/Replace operations: Can be slow on entire document

Architecture Evolution

The scripting subsystem has undergone two major refactorings to reach its current state:

Phase 1 (October 2025): Refactored from the original multi-layered architecture to a Command Executor pattern with IScriptCommandExecutor using switch expression pattern matching. See ADR: Refactor Scripting to Command Executor Pattern.

Phase 2 (October 2025): Migrated from monolithic executor to dispatcher pattern with individual handlers:

  • ✅ Replaced IScriptCommandExecutor with IScriptCommandDispatcher
  • ✅ Migrated all 8 command types to individual IScriptCommandHandler<T> implementations
  • ✅ Removed switch expression entirely
  • ✅ Implemented dynamic dispatch via runtime type resolution
  • ✅ Auto-registration of handlers via Scrutor

Result: The subsystem now follows the Open/Closed Principle - new commands can be added without modifying the dispatcher, and each command handler is independently testable and maintainable.
