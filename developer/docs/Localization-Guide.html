Localization Implementation Guide

This document describes the localization infrastructure implemented for the Office Add-ins project, providing guidelines for maintaining and extending localized content.

Overview

The localization system is designed with the following principles:

  • Simplicity: Easy to use and maintain
  • Separation of Concerns: Domain errors separated from UI text
  • Testability: Tests validate error codes, not localized messages
  • Flexibility: Supports parameterized strings and multiple cultures
  • Maintainability: Clear structure for shared vs application-specific resources

Architecture

Core Components

  1. ILocalizationService (Core.Application.Abstractions.Localization)

    • Abstract interface for retrieving localized strings
    • Supports parameterized strings and culture-specific lookups
    • Keeps domain layer independent of localization infrastructure

  2. LocalizedDomainErrors (Core.Application.Common)

    • Replaces static DomainErrors with localized versions
    • Uses dependency injection to access localization service
    • Maintains same API surface as original DomainErrors

  3. Resource Files

    • Infrastructure.Common/Localization/DomainErrors.resx - Shared domain error messages
    • WordAddIn/Localisation/Resources.resx - Word-specific UI text and dialog messages

  4. WordLocalizationService (WordAddIn.Common)

    • Combines domain errors and UI resources
    • First checks domain errors, then UI resources
    • Provides unified localization for Word Add-in

Resource File Structure

Domain Errors (DomainErrors.resx)

Contains error messages for all domain operations, using error codes as keys:

<data name="Document.NotFound" xml:space="preserve">
  <value>Document not found.</value>
</data>
<data name="Document.OpenDocumentError" xml:space="preserve">
  <value>Error Opening Document with Id: {0}</value>
</data>

UI Resources (Resources.resx)

Contains user interface text, dialogs, and Ribbon labels:

<data name="UI.BrandTitle" xml:space="preserve">
  <value>K+B Add-in</value>
</data>
<data name="UI.SearchRestOfDocument" xml:space="preserve">
  <value>Do you want to search the rest of the document?</value>
</data>
<data name="Ribbon.Tab.KutanaBrochet" xml:space="preserve">
  <value>K+B</value>
</data>

Usage Patterns

Using Localized Domain Errors

public class MyHandler : ICommandHandler<MyCommand, Result>
{
    private readonly LocalizedDomainErrors _errors;
    
    public MyHandler(LocalizedDomainErrors errors)
    {
        _errors = errors;
    }
    
    public Result Handle(MyCommand command, CancellationToken cancellationToken)
    {
        // Simple error
        if (condition)
        {
            return Result.Failure(_errors.Document.NotFound);
        }
        
        // Parameterized error
        var error = _errors.Document.OpenDocumentError(documentId);
        return Result.Failure(error);
    }
}

Using UI Localization

public class UserNotifier : IUserNotifier
{
    private readonly ILocalizationService _localization;
    
    public UserNotifier(ILocalizationService localization)
    {
        _localization = localization;
    }
    
    private string BrandedTitle(string title)
    {
        if (string.IsNullOrWhiteSpace(title))
        {
            return _localization.GetString("UI.BrandTitle");
        }
        
        return _localization.GetString("UI.BrandTitleWithContext", title);
    }
}

Dependency Injection Setup

In ThisAddIn.cs:

builder.Services
    .AddLocalization(options => options.ResourcesPath = "Localisation")
    .AddSingleton<ILocalizationService, WordLocalizationService>()
    .AddSingleton<LocalizedDomainErrors>()
    .AddSingleton<LocalizedErrorFactory>();

Testing Guidelines

✅ Correct: Test Error Codes

[Test]
public void Should_Return_Correct_Error_Code()
{
    // Act
    var result = handler.Handle(command);
    
    // Assert
    Assert.True(result.IsFailure);
    Assert.Equal("Document.NotFound", result.Error.Code);
}

❌ Incorrect: Test Error Messages

[Test]
public void Should_Return_Error_Message() // DON'T DO THIS
{
    // Act
    var result = handler.Handle(command);
    
    // Assert
    Assert.Equal("Document not found.", result.Error.Message); // Brittle!
}

Testing Localization Services

[Test]
public void Should_Use_Localization_Service()
{
    // Arrange
    var mockLocalization = Substitute.For<ILocalizationService>();
    mockLocalization.GetString("Document.NotFound").Returns("Custom message");
    
    // Act & Assert
    mockLocalization.Received(1).GetString("Document.NotFound");
}

Adding New Localized Content

1. Domain Errors

  1. Add error code and message to DomainErrors.resx
  2. Add property to appropriate error class in LocalizedDomainErrors.cs
  3. Test using error code, not message

2. UI Text

  1. Add key-value pair to Resources.resx
  2. Update Resources.Designer.cs if needed
  3. Use through ILocalizationService.GetString()

3. Parameterized Strings

Use {0}, {1}, etc. for parameters:

<data name="UI.ReplacementComplete" xml:space="preserve">
  <value>Replaced {0} items in {1} seconds</value>
</data>

var message = _localization.GetString("UI.ReplacementComplete", count, elapsed);

Localization for Other Office Applications

When adding Excel or PowerPoint add-ins:

  1. Create application-specific resource files (e.g., ExcelAddIn/Localisation/Resources.resx)
  2. Create application-specific localization service (e.g., ExcelLocalizationService)
  3. Register in dependency injection container
  4. Domain errors remain shared across all applications

Culture and Language Support

Current Support

  • Default culture (English)
  • Automatic fallback to key if resource not found
  • Support for culture-specific localization through ILocalizationService

Adding New Languages

  1. Create culture-specific resource files (e.g., Resources.fr-FR.resx)
  2. Translate all values, keeping keys identical
  3. .NET will automatically select appropriate culture based on user settings

Encoding and Character Support

  • Resource files use UTF-8 encoding
  • Supports diacritics and special characters
  • Automatic encoding handling by .NET resource system

Best Practices

  1. Use Error Codes as Keys: This ensures consistency and makes mapping clear
  2. Fallback Strategy: Always return the key if localization fails
  3. Parameterized Strings: Use String.Format pattern for dynamic content
  4. Test Error Codes: Never test localized messages in unit tests
  5. Shared vs Specific: Keep domain errors shared, UI text application-specific
  6. Resource Organization: Group related resources with prefixes (UI., Ribbon., Dialog.)

Maintenance

Updating Messages

  1. Modify resource files (.resx)
  2. Update Designer files if adding new entries
  3. No code changes required for message text updates

Adding Features

  1. Define error codes in domain layer
  2. Add localized messages to resource files
  3. Use through LocalizedDomainErrors or ILocalizationService

Debugging

  • If localization fails, the service returns the key as fallback
  • Enable localization debugging through IStringLocalizer if needed
  • Use culture-specific testing to verify correct resource loading

Migration from Hardcoded Strings

When updating existing code:

  1. Identify hardcoded strings
  2. Define appropriate resource keys
  3. Add to relevant resource file
  4. Update code to use localization service
  5. Update tests to validate error codes instead of messages

This maintains backward compatibility while enabling localization.
